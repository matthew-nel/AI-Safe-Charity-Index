# Purpose

This document defines the editorial calendar for the AI Safe Use Knowledge Base for UK Charities.

It coordinates:

- monthly research
- public guidance updates
- release notes
- website content
- LinkedIn content
- newsletter or Substack content
- research participation
- quarterly assurance
- annual reporting

The calendar exists to keep the project active without creating an unrealistic publishing burden.

The core principle is:

> **Research first, publish what matters, and reuse each strong insight across the right channels.**

---

# Editorial Goals

The editorial system should:

- keep the knowledge base current
- answer real charity questions
- explain important changes
- build trust
- invite research participation
- support public learning
- create a consistent publishing rhythm
- avoid low-value content for its own sake
- turn one strong update into several useful formats
- maintain editorial independence

# Editorial Rhythm

## Weekly

One useful public output where possible.

Examples:

- LinkedIn post
- short website update
- research question
- practical scenario
- myth-busting post
- new template preview

Weekly publishing is optional.

Do not publish weak content to maintain frequency.

## Monthly

Minimum monthly outputs:

- one knowledge-base release or no-change review
- one release summary
- one LinkedIn release post
- one newsletter or Substack update
- one research or feedback invitation
- updated changelog
- updated roadmap status

## Quarterly

Outputs:

- quarterly assurance summary
- evidence and regulator briefing
- roadmap update
- What Charities Told Us summary
- accessibility and quality update
- next-quarter priorities

## Annual

Potential outputs:

- annual impact summary
- State of AI in UK Charities report
- annual evidence review
- project strategy update
- contributor acknowledgements
- annual roadmap

# Monthly Operating Cycle

## Week 1 — Research and Listening

Activities:

- collect regulator updates
- collect provider updates
- review charity-sector reports
- review GitHub issues
- review feedback forms
- review research data
- review failed searches
- identify recurring questions

Outputs:

- research intake list
- initial change-impact summary
- possible editorial themes
- research question for the month

## Week 2 — Analysis and Planning

Activities:

- process sources
- update Source Register
- update Claim Register
- review Regulatory Watchlist
- identify affected pages
- prioritise changes
- select monthly editorial theme

Outputs:

- content-impact report
- release scope
- editorial brief
- draft content list

## Week 3 — Drafting and Review

Activities:

- update Markdown
- draft new guidance
- update templates
- prepare release notes
- draft newsletter
- draft LinkedIn content
- complete human review
- seek specialist review where needed

Outputs:

- release candidate
- newsletter draft
- social content
- validation checklist

## Week 4 — Publish and Learn

Activities:

- publish GitHub release
- deploy website
- publish changelog
- publish release summary
- publish LinkedIn post
- send newsletter
- invite feedback
- monitor issues

Outputs:

- public release
- public communication
- feedback collection
- next-month carry-over

Every monthly cycle may produce:

1. One release or documented no-change review
2. One website update
3. One newsletter
4. Two to four LinkedIn posts
5. One research invitation
6. One practical workflow post
7. One evidence or tool update

- release

# Repurposing Model

One strong piece of research should be reused carefully.

Example:

```text
New ICO guidance
      ↓
Knowledge-base update
      ↓
Release note
      ↓
LinkedIn summary
      ↓
Newsletter explanation
      ↓
Trustee briefing note
      ↓
FAQ update
```

Do not copy the same wording across every channel.

Adapt it to:

- audience
- context
- length
- action

# LinkedIn Content Types

Use a mix of:

## Practical Guidance

Example:

> Before you paste meeting notes into AI, ask whether the meeting included HR, safeguarding, health or confidential board information.

## Myth Busting

Example:

> Removing someone's name does not automatically make the information anonymous.

## Project Updates

Example:

> This month's release adds new guidance for volunteers using personal AI accounts.

## Research Questions

Example:

> What is the biggest thing stopping your charity from creating an AI policy?

## What Charities Told Us

Example:

> Small charities are not asking for more AI theory. They are asking what they can safely paste.

## Behind the Scenes

Example:

> The Source Register is how this project connects public guidance back to the evidence behind it.

## Tool Updates

Example:

> A provider feature changed. Here is what charities should review.

# Priority Rules

Prioritise content that:

- corrects unsafe guidance
- responds to regulator change
- answers repeated questions
- supports a major workflow
- reduces confusion
- supports accessibility
- invites useful research
- improves core navigation
- supports the public beta

Deprioritise content that:

- repeats general AI news
- exists only for reach
- promotes tools without a charity need
- creates high maintenance
- lacks evidence
- distracts from core guidance

---

# Fast-Changing Content

Tool and provider content should:

- show a review date
- link official documentation
- state plan and account differences
- avoid permanent claims
- trigger monthly review

Do not create social content from a provider announcement before checking the practical impact.

---

# Evergreen Content

Evergreen topics include:

- Before You Paste
- human review
- data minimisation
- safeguarding
- trustee accountability
- accessibility
- evidence standards

Evergreen does not mean never reviewed.

---

# Editorial Independence

Content should not be shaped by:

- affiliate commission
- sponsorship
- vendor pressure
- consultancy prospects
- product access

If a relationship exists:

- disclose it
- keep it separate from safety ratings
- apply the same evidence standard
- provide non-affiliate links where practical

# Definition of Done

A monthly editorial cycle is complete when:

- the research review is complete
- the release is published or no-change review recorded
- changelog is updated
- release summary is public
- LinkedIn content is published where useful
- newsletter is sent where planned
- feedback and research are invited
- results are reviewed
- next-month priorities are set
- unfinished work is carried over visibly

---

# Success Criteria

The editorial calendar succeeds when:

- the knowledge base stays current
- public content answers real charity questions
- research and publishing reinforce one another
- one strong insight is reused efficiently
- the workload remains realistic
- channels support the knowledge base rather than distract from it
- charities know what changed and what to do
- the project builds trust through consistent, useful communication
