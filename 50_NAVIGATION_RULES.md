# Purpose

This document defines how navigation should work across the AI Safe Use Knowledge Base for UK Charities.

The navigation system should help people find practical guidance quickly, even when they do not know the correct technical or legal term.

The primary navigation principle is:

> **Organise around what charities are trying to do, not around how AI technology is categorised.**

---

# Navigation Goals

Navigation should help users:

- understand where to start
- find task-based guidance
- move between overview and detail
- find guidance for their role
- identify charity-size differences
- recognise risk level
- return to related guidance
- find templates and evidence
- report missing or unclear content

---

# Primary Audience

The main navigation should prioritise:

- small charities
- medium-sized charities
- non-technical users
- staff with limited time
- trustees
- volunteers
- teams without specialist support

Advanced guidance for large and complex organisations should remain available without dominating the main experience.

# Charity-Size Navigation

The site should support:

## Small Charities

Prioritise:

- simple first rules
- low-cost controls
- one named owner
- short policy
- small approved-tools list
- practical templates

## Medium-Sized Charities

Prioritise:

- cross-team consistency
- role-based guidance
- formal approval
- AI-use registers
- quarterly review

## Large and Complex Charities

Prioritise:

- procurement
- system inventory
- formal assurance
- DPIAs
- monitoring
- audit
- enterprise governance

The small and medium routes should remain the default.

---

# Task-Based Navigation

Use question-based titles.

Prefer:

- Can I use AI to write a funding bid?
- Can volunteers use ChatGPT?
- Can AI summarise meeting notes?
- Can I paste survey responses into AI?
- What should trustees approve?
- How do we choose an AI tool?

Avoid:

- AI transformation
- Enterprise capability
- Operational innovation
- Governance architecture
- LLM strategy

---

# Risk-Based Navigation

High-risk pages should visibly link to:

- Before You Paste
- GDPR guidance
- Safeguarding guidance
- Incident process
- Specialist review
- Trustee or leadership oversight

Use clear labels:

- Green — Go ahead with normal checks
- Amber — Pause and check
- Red — Stop

Do not use colour alone.

---

# Page-Level Navigation

Every substantive page should include:

- breadcrumbs
- table of contents where useful
- previous and next links
- related guidance
- related templates
- related evidence
- feedback link
- back-to-section link

---

# Breadcrumbs

Use descriptive breadcrumbs.

Example:

```text
Home
› Everyday Charity Work
› Meetings
› Can I upload meeting notes?
```

Avoid exposing repository structure to users.

---

# Sidebar Navigation

The sidebar should:

- show the current section
- highlight the current page
- use short labels
- group related pages
- remain manageable on mobile
- avoid excessive nesting
- show advanced content separately

Recommended maximum depth:

- three visible levels

---

The footer should include:

- About the Project
- About the Creator
- Contact

- Code of Conduct
- Governance
- Changelog

- GitHub
- LinkedIn

---

# Cross-Linking Rules

Every page should link to:

- at least one related practical page
- at least one deeper explainer where relevant
- a template where relevant
- evidence or sources
- feedback

Avoid linking every possible related page.

Choose the most useful next steps.

---

# Navigation Labels

Labels should:

- use sentence case
- use plain English
- remain stable
- describe user intent
- avoid internal terminology

Good:

- Before you paste
- Personal data
- AI for trustees
- Approved tools

Poor:

- Framework 01
- Domain 2
- Governance artefacts
- LLM operations

---

# Link Text

Use descriptive link text.

Good:

- Read the personal data guide
- Download the AI policy template
- See the latest release notes

Avoid:

- Click here
- Learn more
- Read this
- More

---

# Navigation and Search

Navigation and search should work together.

If users repeatedly search for a term that is not in navigation:

- add a synonym
- improve a title
- create an FAQ
- review whether a new page is needed

Do not use search as an excuse for poor navigation.

---

# Navigation Testing

Test navigation with tasks such as:

- Find out whether you can upload meeting notes.
- Find a short AI policy template.
- Find guidance for trustees.
- Find the ChatGPT guide.
- Find what personal data means.
- Report outdated guidance.
- Join current research.

Observe:

- where users hesitate
- which labels are misunderstood
- whether they backtrack
- whether they rely on search
- whether the right page appears

---

# Mobile Navigation

On mobile:

- keep the main menu short
- use a clear menu button
- preserve current section context
- make tap targets large
- avoid deep accordion chains
- keep breadcrumbs readable
- ensure search is easy to find

---

# Navigation Accessibility

Navigation should:

- work with keyboard
- expose visible focus
- use semantic landmarks
- have logical reading order
- clearly identify current page
- avoid colour-only indicators
- support screen readers
- not trap focus

---

# Navigation Governance

Navigation changes should be reviewed when they affect:

- top-level sections
- URLs
- learning paths
- audience routes
- search
- templates
- release pages

Major navigation changes may require a minor or major version update.

---

# Navigation Change Checklist

Before changing navigation:

- [ ] User need is clear.
- [ ] Existing search data was reviewed.
- [ ] Existing pages were checked for duplication.
- [ ] New labels use plain English.
- [ ] URLs remain stable where possible.
- [ ] Redirects are prepared if needed.
- [ ] Mobile behaviour was reviewed.
- [ ] Accessibility was checked.
- [ ] Breadcrumbs were updated.
- [ ] Sidebars were updated.
- [ ] Related links were updated.
- [ ] Search index was updated.
- [ ] Release notes explain the change.

---

# Monthly Navigation Review

Review:

- failed searches
- high-exit pages
- repeated feedback
- new content
- orphaned pages
- duplicate labels
- dead-end pages
- mobile issues
- accessibility issues

---

# Quarterly Navigation Review

Review:

- top-level architecture
- role-based pathways
- charity-size pathways
- section balance
- search-navigation overlap
- advanced content separation
- roadmap alignment

---

# Success Criteria

Navigation succeeds when:

- users find the right page quickly
- non-technical readers understand the labels
- small and medium-sized charities feel prioritised
- role-based routes help without duplicating content
- high-risk guidance is easy to reach
- mobile navigation remains simple
- search supports rather than replaces navigation
- the site can grow without becoming confusing
