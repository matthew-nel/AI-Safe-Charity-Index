# Changelog

This file records significant changes to the AI Safe Use Knowledge Base for UK Charities.

The changelog exists to make the project transparent.

It should help readers understand:

- what changed
- when it changed
- why it changed
- which pages were affected
- which evidence was reviewed
- whether users need to take action

The changelog should not replace detailed release notes.

Release notes explain one release in depth.

The changelog provides a continuous record across the life of the project.

---

# Changelog Principles

Every material change should be:

- dated
- versioned
- described clearly
- linked to evidence where relevant
- linked to a pull request or release where available
- attributed
- honest about limitations

Do not hide significant corrections.

Do not use vague entries such as:

- updates
- improvements
- fixes
- minor changes

Explain what actually changed.

---

# Versioning Format

Use:

```text
MAJOR.MINOR.PATCH
```

Examples:

- `1.0.0` — first public release
- `1.1.0` — new guidance or templates
- `1.1.1` — correction or clarification
- `2.0.0` — major framework or structural change

---

# Change Categories

Use the following categories.

## Added

New pages, features, templates, workflows or research.

## Changed

Material changes to existing guidance, structure or wording.

## Corrected

Errors, unsupported claims, broken links or inaccurate examples.

## Deprecated

Content that remains available temporarily but should no longer be used as the primary guidance.

## Removed

Content removed from the active knowledge base.

## Security

Changes connected to security, privacy or exposure of sensitive information.

## Accessibility

Changes improving accessibility or correcting accessibility barriers.

## Evidence

New sources, claim changes, evidence-level changes or research updates.

## Research

Survey findings, interview findings, usability research or new research methods.

## Governance

Changes to ownership, review, contributor roles or decision rights.

## Technical

Build, search, deployment, validation or repository changes.

# Correction Policy

A correction should be recorded when:

- a factual statement was wrong
- a legal or regulatory summary was misleading
- a safety classification was inappropriate
- a tool feature was described incorrectly
- a source did not support the published claim
- a significant link was wrong
- a template could lead to unsafe practice

Minor spelling and formatting changes do not always need an individual changelog entry.

They may be grouped into a patch release.

---

# Silent Changes

Avoid silent changes to:

- legal interpretation
- GDPR guidance
- safeguarding guidance
- risk classification
- tool recommendation
- trustee responsibilities
- privacy advice
- human-oversight requirements
- significant templates

These changes must be visible in:

- page history
- changelog
- release notes
- version number

---

# Evidence Changes

Record when:

- a new primary source is added
- a source is superseded
- evidence confidence changes
- sources conflict
- a claim is withdrawn
- original research changes a recommendation

Suggested wording:

> Changed the evidence level from Moderate to High after reviewing updated ICO guidance and two supporting charity-sector sources.

---

# Deprecation Entries

A deprecation entry should explain:

- what is deprecated
- why
- replacement guidance
- date of final removal where known
- whether charities need to act

Example:

> Deprecated the previous AI policy template because it did not clearly distinguish safeguarding information from other personal data. Use version 2.0 instead.

---

# Removed Content

Record removal when:

- guidance is unsafe
- duplication creates confusion
- evidence no longer supports the page
- a provider product has been withdrawn
- a template is replaced
- a URL cannot be maintained

Where appropriate, preserve an archived copy.

---

# Research Entries

Research changelog entries should explain:

- question investigated
- method
- sample
- date
- major findings
- limitations
- pages changed because of the research

Example:

> Updated volunteer guidance after interviews with 12 small charities identified frequent use of personal ChatGPT accounts for internal work.

Do not imply that small research samples represent the whole sector.

---

# Accessibility Entries

Record material accessibility changes such as:

- keyboard-navigation fixes
- contrast corrections
- screen-reader improvements
- heading corrections
- accessible form changes
- table redesign
- PDF remediation
- plain-English improvements where they materially affect access

---

# Contributor Acknowledgements

Acknowledge accepted contributions where appropriate.

Possible formats:

- GitHub username
- name
- organisation
- specialist reviewer role
- research participant group

Do not identify research participants without permission.

# Changelog Ownership

The primary maintainer is responsible for ensuring the changelog is updated before every release.

The release should not be considered complete until:

- the version is added
- the date is correct
- material changes are recorded
- corrections are visible
- known limitations are recorded
- contributor acknowledgements are included where appropriate
