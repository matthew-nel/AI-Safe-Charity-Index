# Purpose

This document defines the accessibility standard for the AI Safe Use Knowledge Base for UK Charities.

Accessibility applies to:

- website content
- navigation
- search
- forms
- templates
- downloads
- diagrams
- tables
- videos
- audio
- code examples
- AI-generated outputs
- contributor workflows

The project should aim to meet **WCAG 2.2 AA** wherever practical.

Accessibility is not an optional finishing step.

It is part of editorial quality, technical quality and public trust.

---

# Core Principle

> **Make the safest guidance usable by the widest possible range of people.**

A page is not useful if people cannot read, navigate, understand or act on it.

---

# Accessibility Goals

The project should support people who may:

- use a screen reader
- navigate by keyboard
- use voice control
- have low vision
- have colour-vision differences
- have dyslexia
- have cognitive or learning differences
- experience fatigue
- use a small screen
- use older devices
- have low digital confidence
- speak English as an additional language
- need plain-English explanations

---

# Standards

The project should use:

- WCAG 2.2 AA as the main benchmark
- semantic HTML
- accessible Markdown structures
- plain English
- progressive enhancement
- inclusive content design
- manual testing as well as automated checks

Automated tools alone are not enough.

---

# Content Accessibility

## Headings

Use headings in a logical order.

Good:

```text
H1
  H2
    H3
  H2
```

Avoid:

```text
H1
  H4
  H2
```

Each page should have one clear main heading.

## Paragraphs

Use:

- short paragraphs
- one main idea per paragraph
- clear topic sentences
- direct language

Avoid large blocks of dense text.

## Lists

Use lists where they improve scanning.

Do not turn every sentence into a bullet.

## Plain English

Prefer:

- common words
- active voice
- short sentences
- clear instructions
- practical examples

Explain:

- acronyms
- technical terms
- legal terms
- AI terminology

## Reading Level

Write for understanding rather than expertise.

Do not make legal or technical ideas inaccurate in the name of simplicity.

---

# Links

Link text should describe the destination.

Good:

- Read the personal data guide
- Download the AI policy template
- See the latest release notes

Avoid:

- Click here
- Read more
- This page
- More

Links should be understandable out of context.

---

# Colour

Do not use colour as the only way to communicate meaning.

Green, Amber and Red components must also include:

- text label
- icon
- explanation
- action

Colour contrast should meet WCAG 2.2 AA requirements.

---

# Typography

Use:

- readable font sizes
- generous line spacing
- clear hierarchy
- reasonable line length
- fonts that remain legible at zoom

Avoid:

- very small text
- thin text on light backgrounds
- long centred paragraphs
- decorative fonts for body copy
- all-caps body text

---

# Keyboard Navigation

All interactive elements should be usable by keyboard.

Users should be able to:

- reach navigation
- open menus
- use search
- submit forms
- activate buttons
- move through decision trees
- close dialogs
- follow links

Focus should remain visible.

Do not trap keyboard focus.

---

# Screen Readers

Use:

- semantic landmarks
- correct heading order
- descriptive labels
- meaningful alt text
- accessible form labels
- clear status messages
- correct table markup

Test important user journeys with a screen reader.

---

# Images

Every meaningful image should have alt text.

Alt text should:

- explain the purpose
- be concise
- avoid repeating nearby text
- avoid starting with “image of”
- include important text shown in the image

Decorative images should use empty alt text where appropriate.

---

# Diagrams and Decision Trees

Every diagram should have:

- a text equivalent
- a logical reading order
- meaningful labels
- no colour-only meaning
- accessible downloadable version where useful

The Before You Paste decision tree must also exist as plain text.

---

# Tables

Use tables only for genuine tabular information.

Requirements:

- clear headers
- simple structure
- no merged cells where avoidable
- short cell content
- mobile-friendly presentation
- text alternative where the table is complex

Do not use tables for page layout.

---

# Forms

Forms should include:

- visible labels
- clear instructions
- useful error messages
- keyboard access
- logical tab order
- accessible validation
- clear privacy information
- confirmation after submission

Do not rely on placeholder text as the only label.

---

# Search

Search should:

- be easy to locate
- work with keyboard
- expose result counts
- announce updates to screen readers
- support plain-English queries
- remain usable at high zoom
- work on mobile

---

# Navigation

Navigation should:

- use semantic landmarks
- show current location
- support keyboard access
- provide visible focus
- use clear labels
- avoid deep nesting
- remain usable on mobile
- include a skip link

---

# Downloads

Downloads should be accessible.

Prefer:

- HTML or Markdown first
- accessible DOCX where needed
- tagged PDF only when necessary
- structured headings
- readable tables
- accessible links
- meaningful filenames

Do not publish image-only PDFs.

---

# PDF Standard

If PDFs are offered:

- provide an HTML equivalent where possible
- use tagged structure
- include document title
- set reading order
- add alt text
- use accessible tables
- define language
- verify bookmarks
- test with assistive technology

---

# Video

Videos should include:

- captions
- transcript
- accessible controls
- clear audio
- no essential information conveyed visually only

Avoid rapid flashing or unnecessary motion.

---

# Audio

Audio should include:

- transcript
- speaker identification where useful
- accessible playback controls
- downloadable transcript where practical

---

# Motion and Animation

Animation should:

- be limited
- never block access to content
- respect reduced-motion settings
- avoid flashing
- avoid unnecessary auto-play
- provide controls where needed

---

# Mobile Accessibility

On mobile:

- use large tap targets
- avoid horizontal scrolling
- preserve readable text
- keep forms simple
- make menus easy to close
- ensure tables can be understood
- avoid sticky elements covering content

---

# Zoom and Reflow

The site should remain usable when:

- browser zoom is increased
- text size is increased
- the viewport is narrow
- users reflow content

Do not require horizontal scrolling for normal reading.

---

# Language

Use:

- inclusive language
- respectful terminology
- person-centred wording
- precise descriptions
- consistent terms

Avoid:

- defining people only by vulnerability
- stigmatising language
- unnecessary gender assumptions
- jargon-heavy role labels
- idioms that may confuse

---

# Cognitive Accessibility

Support understanding by:

- showing the answer early
- using consistent page structure
- limiting competing calls to action
- using clear labels
- explaining consequences
- separating essential from advanced detail
- avoiding unnecessary complexity
- making next steps explicit

---

# AI-Generated Content

AI-generated content must not be published without accessibility review.

Check:

- heading structure
- link text
- reading level
- unnecessary jargon
- table complexity
- image descriptions
- assumptions about users
- tone
- logical sequence

AI may produce grammatically correct content that is still difficult to understand.

---

# Accessibility of Templates

Templates should:

- use clear headings
- include instructions
- avoid inaccessible tables
- use meaningful filenames
- work with keyboard
- avoid colour-only status
- include plain-text alternatives
- explain optional and required fields

---

# Accessibility of Traffic-Light Guidance

Green, Amber and Red guidance should include:

```text
Green — Go ahead with normal checks
Amber — Pause and check
Red — Stop
```

Each callout should explain:

- what the label means
- why it applies
- what to do next

---

# Accessibility Statement

The website should publish an accessibility statement covering:

- accessibility commitment
- standard targeted
- known issues
- contact route
- response process
- review date
- enforcement information where relevant

---

# Reporting Accessibility Issues

Users should be able to report:

- keyboard problems
- screen-reader issues
- contrast problems
- inaccessible forms
- unclear language
- inaccessible downloads
- mobile issues
- missing alt text
- difficult tables

Reports should request:

- page URL
- device
- browser
- assistive technology where relevant
- description
- impact

---

# Accessibility Issue Priority

Use:

## Critical

- user cannot access core safety guidance
- user cannot submit a form
- keyboard trap
- essential content unavailable to screen readers

## High

- important workflow difficult to complete
- serious contrast issue
- inaccessible decision tree
- inaccessible download without alternative

## Medium

- poor heading order
- unclear link text
- difficult table
- missing non-critical alt text

## Low

- minor wording
- cosmetic inconsistency
- small non-blocking layout issue

---

# Testing Approach

Use both automated and manual testing.

## Automated

Possible checks:

- Axe
- Lighthouse
- Pa11y
- HTML validation
- contrast checks

## Manual

Test:

- keyboard only
- screen reader
- zoom
- reflow
- mobile
- reduced motion
- form errors
- decision trees
- search
- downloads

---

# Core User Journeys to Test

1. Find Before You Paste.
2. Use the decision tree.
3. Find the AI policy template.
4. Search for meeting notes.
5. Read a tool guide.
6. Report an error.
7. Join research.
8. Navigate on mobile.
9. Download a template.
10. Find the accessibility statement.

---

# Accessibility Review Frequency

## Every Pull Request

Check:

- headings
- links
- alt text
- form changes
- interactive changes

## Monthly

Review:

- new pages
- new downloads
- reported issues
- automated scan results

## Quarterly

Review:

- navigation
- search
- templates
- mobile experience
- major user journeys
- known issues

## Annually

Conduct a deeper audit.

Pages may include:

```yaml

accessibility_review_date: "2026-07-12"
known_accessibility_issues: []
```

Do not use metadata as a substitute for actual testing.

---

# Contributor Requirements

Contributors should:

- follow heading rules
- use descriptive links
- add alt text
- test keyboard access for interactive changes
- avoid colour-only meaning
- use plain English
- document known issues

---

# Release Requirements

A release should not proceed where:

- core navigation is inaccessible
- important forms cannot be used
- the decision tree has no text alternative
- critical content is unavailable at high zoom
- new downloads are inaccessible without an alternative
- known critical issues are unresolved

---

Before publication:

- [ ] Heading order is logical.
- [ ] Links are descriptive.
- [ ] Images have appropriate alt text.
- [ ] Colour is not the only indicator.
- [ ] Contrast meets the target.
- [ ] Keyboard navigation works.
- [ ] Focus is visible.
- [ ] Forms have labels.
- [ ] Error messages are clear.
- [ ] Tables use headers.
- [ ] Mobile layout works.
- [ ] Zoom and reflow work.
- [ ] Decision trees have text alternatives.
- [ ] Downloads are accessible.
- [ ] Captions or transcripts exist where needed.

# Claude Prompt

Use:

```text
Review the selected page or release against the Accessibility Standard.

Assess:

- headings
- plain English
- link text
- images and alt text
- tables
- forms
- navigation
- keyboard access
- colour use
- mobile layout
- cognitive load
- downloads
- video and audio
- AI-generated content

Produce:

1. Critical issues.
2. High-priority issues.
3. Medium-priority issues.
4. Low-priority issues.
5. Exact recommended changes.
6. Content revisions.
7. Technical checks requiring human testing.
8. Known limitations.

Do not claim WCAG compliance based only on automated review.
```

---

# Success Criteria

The accessibility standard succeeds when:

- people can access core safety guidance regardless of how they browse
- non-technical users understand the content
- disabled users can complete key tasks
- accessibility is considered before release
- reported issues are resolved transparently
- templates and downloads remain usable
- AI-generated content does not introduce barriers
- accessibility improves continuously rather than being treated as a one-off audit
